Selected transforms:

1. **Object transform** — the real object is not a mission document or prompt

2. **Unit transform** — the unit of optimization changes from one run or

## Central Risk Surface: Publish / Promote / Rollback

The publish/adoption/promotion/rollback components are probably the largest
systemic risk in Choir. They are where candidate work becomes active reality,
where user trust can be lost, where foreground-tail changes can be overwritten,
where evidence can be laundered into authority, and where a platform deploy can
be mistaken for user-computer promotion.

Therefore Campaign Compiler v0 should center on filing down this transition
surface before it optimizes for breadth, autonomy, or 24/7 concurrency.

The first campaign should repeatedly exercise small but real transition cases:

- publish a tiny package with complete provenance and evidence;
- reject a package with missing source delta, missing verifier evidence, or
  missing rollback refs;
- adopt into one owner-controlled active computer;
- verify recipient build and artifact digests before promotion;
- promote only after owner/platform review is represented as product evidence;
- roll back the promoted adoption and prove active state returns to the prior
  refs;
- roll forward again from the same verified package when allowed;
- attempt adoption across a second divergent computer and record portability
  blockers precisely;
- keep foreground-tail changes intact during candidate verification and
  promotion;
- synthesize run acceptance from existing Trace/VText/package/adoption evidence
  rather than from a manually seeded success record.

This makes Campaign Compiler less vulnerable to execution-order perturbations.
The campaign is not "build a campaign compiler UI, then later test promotion."
The campaign is "use publish/promote/rollback as the load-bearing rehearsal
space for campaign compilation." UI, orchestration, transforms, and evidence
schema should be judged by whether they make these transitions safer and more
legible.

Edge-case inventory for the campaign:

- package published with no meaningful source delta;
- package has source delta but no human-readable evidence;
- verifier passes stale candidate refs;
- verifier checks source but not rebuilt runtime/UI artifacts;
- recipient active computer changes during verification;
- owner approves a package whose verifier evidence is incomplete;
- promotion succeeds but rollback profile is missing or invalid;
- rollback succeeds but roll-forward uses stale artifacts;
- divergent computer cannot apply the package cleanly;
- two candidates race for the same active computer;
- a background worker reports success without product-visible package refs;
- Trace/VText disagree on package or adoption status;
- run acceptance claims `promotion-level` without adoption verifier evidence;
- platform deploy evidence is accidentally treated as user-computer promotion.

## Bespoke Per-User Adaptation Evals

User-computer promotion is not a normal package pull. When user A publishes an
app or feature, user B may need a modified descendant that fits B's current
apps, data, preferences, model policy, runtime capabilities, visual taste,
privacy boundaries, and active source refs. Campaign Compiler should treat this
as adaptive transplantation, not installation.

The eval object is therefore:

```text
PublishedPackage
-> RecipientAdaptationPlan
-> RecipientCandidate
-> verifier suite
-> human/video QA packet when needed
-> adoption
-> promotion
-> rollback / roll-forward
-> reentry digest
```

Best-practice eval ideas to import:

- scenario taxonomy: enumerate what kinds of adaptation can happen and make
  missing scenario coverage explicit;
- held-out and adversarial cases: include packages expected to fail, packages
  requiring local modification, and packages that should be rejected despite
  looking useful;
- trajectory logging: preserve raw action traces, tool calls, package refs,
  verifier outputs, screenshots/video, and owner decisions for replay;
- separation of hard gates and soft scores: hard invariants decide whether a
  transition may proceed; scores rank quality inside the admissible space;
- regression replay: every surprising failure becomes a replayable eval case
  before the campaign widens;
- production-shaped evals: use the same product APIs, auth, VText, Trace,
  adoption, and rollback paths that staging/production will use.

Adaptation scenario taxonomy:

- clean adoption: package applies to recipient with no local changes;
- local configuration: package needs recipient settings, theme, model policy,
  or app layout changes;
- source adaptation: package needs code changes to fit recipient runtime or UI;
- data/privacy adaptation: package behavior must change because recipient data
  or sharing policy differs;
- dependency adaptation: recipient lacks or has a different dependency/runtime;
- conflict adaptation: recipient active source changed since package publish;
- taste adaptation: app works but needs user-specific UX/content behavior;
- rejection: package is unsuitable and should produce a clear blocker, not a
  weak adaptation.

Verifier/eval layers:

1. **Static gates**: source delta exists, manifest valid, provenance present,
   owner/visibility policy valid, no forbidden route or secret leakage.
2. **Build gates**: recipient candidate rebuilds runtime/UI artifacts from the
   adapted source with recorded digests.
3. **Behavior gates**: focused tests and product-path smoke prove the adapted
   feature works in the recipient computer.
4. **State safety gates**: foreground-tail accounting, active ref freshness,
   rollback profile, route/default-base changes, and no lost updates.
5. **Human QA gates**: screenshots or short videos prove user-visible behavior
   when visual judgment matters.
6. **Longitudinal gates**: post-promotion observation, rollback, roll-forward,
   reentry digest, and regression replay.

Video should be first-class evidence when human QA is needed. It is not a
replacement for automated checks; it is a compact human-review artifact that
lets the system avoid asking the owner to reconstruct behavior from logs.

Video QA packet:

```yaml
video_qa_packet:
  scenario_id:
  package_id:
  adoption_id:
  before_ref:
  candidate_ref:
  active_ref_after_promotion:
  video_ref:
  screenshot_refs:
  transcript_or_chapters:
  what_to_notice:
  automated_checks:
  human_decision_needed:
  rollback_ref:
```

The long-term learning loop is that video/human review outcomes become labels
for future automated evals. If the human flags "layout wrong", "wrong tone",
"feature works but not for my workflow", or "unsafe state change", that becomes
a scenario label and a future verifier target.

## Progress Criterion

Move uphill when Choir can:

- attach user intent to a durable campaign rather than a transient prompt;
- compile mission mass correctly instead of overstructuring tiny work or
  undercontrolling risky work;
- select cognitive transforms by default, record why they were selected, and
  evaluate whether they changed route, verifier, evidence, scope, or stopping
  condition;
- emit MissionGeometry and WorkOrders that include real capability envelopes;
- keep candidate-computer mutation isolated from active computers;
- return EvidencePackets with claims, commands or observations, artifact refs,
  risks, blockers, and next probes;
- make VText show the current campaign, mission, active work orders, latest
  evidence, blockers, next move, and status;
- publish source-changing work as AppChangePackages and repeatedly move small
  packages through adoption/build/verify/promote/rollback/roll-forward until
  edge cases become explicit tests or documented blockers;
- preserve campaign memory so the next mission is selected from evidence, not
  from a fresh blank prompt;
- eventually run concurrent heterogeneous construction/research/publication
  work without losing authority boundaries or overwhelming human review.

Move downhill when:

- Campaign Compiler becomes a prompt template with no persisted objects;
- the system creates mission paperwork but no candidate/promotion evidence;
- simple tasks become bureaucratic long missions;
- risky source mutation happens in the active foreground computer;
- multiple agents mutate one canonical state space without isolation;
- verifier output is prose without raw evidence pointers;
- VText becomes a status dump instead of an owner-readable semantic state;
- a cycle creates work faster than it resolves, prunes, promotes, or escalates;
- outer Codex hand-edits around missing product transitions and calls that
  Choir-in-Choir progress.

## Homotopy Axes

Increase realism continuously along these axes:

1. **Representation:** mission doc -> VText-rendered campaign state -> typed
   runtime records -> cached live system model.
2. **Mutation:** no mutation -> tiny candidate state mutation -> source delta
   package -> app/runtime/UI package -> platform-computer promotion.
3. **Computer divergence:** one active/candidate pair -> source/recipient pair
   -> multiple divergent computers -> platform computer/default distribution.
4. **Agent topology:** super-only planning -> one vsuper -> worker/verifier
   cosupers -> multiple vsupers -> heterogeneous cosuper teams.
5. **Time horizon:** one prompt -> bounded mission -> sweep -> fly -> cycle.
6. **Task heterogeneity:** one small implementation -> implementation plus
   verification -> construction plus research -> publication and distribution.
7. **Reentry:** final chat response -> VText report -> email latest VText ->
   radio/public digest.
8. **Transform maturity:** manually requested lenses -> default compiler
   transform selection -> logged invocations -> transform graph with citations
   and outcome evidence.
9. **Perspective plurality:** one owner perspective -> worker/verifier
   perspective split -> researcher/editor/security/reporter perspectives ->
   multi-user/public perspective with source-grounded disagreement.
10. **Runtime plurality:** one web runtime -> worker VM runtime -> candidate
    computer source/runtime/UI builds -> heterogeneous runtimes with explicit
    capability envelopes.

These are not ladder rungs. Every low-resolution proof must preserve the same
identity: candidate mutation, typed evidence, serialized promotion, campaign
memory, and owner-readable reentry.

2. Create one candidate world for a tiny, reversible state/source mutation.
3. Return one EvidencePacket and VText campaign update.
4. Publish an AppChangePackage if source changed.
5. Reject at least one incomplete package/adoption path for a precise reason.
6. Pull/adopt/promote into an owner-controlled active computer with rollback.
7. Roll back and roll forward from recorded refs.
8. Repeat across a second divergent computer to prove package portability.
9. Add a longer mission horizon only after package/adoption proof is stable.
10. Add heterogeneous work orders: construction, verifier, researcher, reporter.
11. Add multiple vsupers only after single-vsuper evidence remains legible.
12. Add multi-user or public-perspective review only after source/adoption
    privacy boundaries are explicit.
13. Move from fly to cycle only when budget, decay, kill conditions, verifier
    disagreement handling, and human attention queue are implemented.

This ramp is a realism schedule, not a substitute for the homotopy invariants.

## Forbidden Shortcuts

- Do not implement Campaign Compiler as only a prompt template.
- Do not optimize this mission around improving Codex `/goal`; the benchmark is
  Choir-in-Choir product-path execution.
- Do not add a decorative campaign dashboard before typed state and evidence
  exist.
- Do not call VText prose a MissionGeometry record unless it is retrievable as
  structured state or explicitly marked as docs-level.
- Do not use `/api/agent/*`, `/api/prompts`, `/api/test/*`, `/internal/*`, raw
  event mutation endpoints, direct service ports, or manual success seeding for
  acceptance.
- Do not use host platform deploy as proof of user-computer promotion.
- Do not publish packages without source delta, provenance, verifier evidence,
  and rollback refs.
- Do not let external research/publication signals directly modify authority,
  tools, canonical state, or campaign invariants.
- Do not start 24/7 cycles before budget, decay, kill switch, verifier, and
  human attention policies are real.
- Do not let super, vsuper, and cosuper collapse into one full-scope agent when
  candidate boundaries matter.

## Rollback Policy

- Platform substrate rollback: revert or follow-up fix through GitHub `main`,
  CI, deploy, health identity, and staging proof.
- Candidate rollback: discard candidate computer or rollback candidate source
  ref; preserve blocker/evidence.
- AppChangePackage rollback: reject, archive, or roll back adoption using the
  product adoption rollback refs.
- Campaign rollback: append corrected campaign state and supersede bad
  MissionGeometry/EvidencePacket records. Do not erase old evidence.
- Cycle rollback: dorm or kill the cycle when health conditions fail; preserve
  bag/evidence snapshot.

## Learning Side-Channel

Write durable learning into:

- this mission doc when the campaign route changes;
- `docs/choir-agentic-depth-canonical.md` when campaign/fly/cycle vocabulary
  changes;
- `docs/mission-geometry.md` when product ontology changes;
- runtime tests when a failure becomes an invariant;
- VText campaign state when product-path proof exists;
- Trace/run acceptance when evidence supports promotion.

Cognitive transforms must be logged as action-changing. A transform invocation
that does not change the next probe, verifier, evidence, scope, or stopping
condition should be treated as commentary, not promoted as useful transform IP.

## Stopping Condition

Stop successfully only when staging product evidence shows:

- `choir-self-development` campaign state exists and is owner-readable in VText;
- a visible prompt is compiled into mission geometry with mission mass,
  invariants, default-selected cognitive transforms, work order(s), evidence
  contract, and stopping state;
- at least one candidate-computer or worker-vsuper mutation is isolated from
  the active computer;
- the worker returns an EvidencePacket with raw evidence refs;
- source-changing work, if present, moves through AppChangePackage,
  pull/adoption/build/verify/promote/rollback;
- Trace and run acceptance connect the campaign, mission, work order, evidence,
  package/adoption refs, blocker or success state, and next bounded mission;
- transform invocation records show what action, verifier, evidence, scope, or
  stopping condition changed;
- the next campaign move is selected from recorded evidence, not from an
  ungrounded fresh prompt.

Use `checkpoint_incomplete` when useful substrate or documentation lands but
the full stopping condition is not met.

Use `blocked_incomplete` only after root-cause probes and route-changing
cognitive transforms show that continuing requires external authority, would
violate an invariant, or would repeat an already-falsified probe without new
evidence.
