# CI SBOM Nested-Nix Sandbox Failure

**Date:** 2026-07-09
**Status:** repaired by `c96c7b49`; successful host-side run observed; lane later paused
**Classification:** substrate
**Mutation class of the exposing change:** orange CI/publishing orchestration

## Problem

The first unified-CI SBOM baseline run attempted to build the flake's
`sbom.x86_64-linux.<package>` derivations. All twelve package SBOMs failed.
The workflow still produced a failure manifest, but it did not create a usable
baseline for later differential reuse.

## Evidence

- Commit: `788251320a6a9180f523859537497e0c45fa0251`
- Workflow run: <https://github.com/choir-hip/go-choir/actions/runs/29065559051>
- Job: `Generate Differential SBOMs` (`86276245984`)
- Duration: 11m37s
- Manifest result: 12 `sbom_build_failed` records, including all 11 required
  packages and optional `obscura`
- Representative builder error:

  ```text
  error: creating directory '/nix/var/nix/profiles': Permission denied
  CRITICAL Specified target is not a nix artifact
  ```

## Root Cause Belief

`flake.nix` implements each SBOM as a `runCommand` derivation that invokes
sbomnix on a package output. sbomnix determines runtime dependencies by invoking
`nix build` for its target. That creates a nested Nix invocation inside the
outer Nix build sandbox. The nested invocation cannot initialize the normal Nix
profile path, so every SBOM derivation fails after its package build completes.

This is a substrate failure in the SBOM execution topology, not twelve
independent package failures.

## Existing Replacement Opportunity

The pre-consolidation CI path already used the viable topology:

1. build the package with Nix on the GitHub runner;
2. run the pinned sbomnix executable on the runner, outside the Nix build
   sandbox;
3. write the resulting CycloneDX JSON into the workflow artifact directory.

The repair should reconnect that host-side generation path while retaining the
new derivation-identity manifest, checksum verification, complete-bundle cache,
and component diff.

## 2026-07-16 Status Correction

The replacement opportunity was implemented by commit
`c96c7b490e5efb9999ae6ab8ee6c8b790387eee3`. The current differential builder
builds each package on the GitHub runner, builds pinned sbomnix from the
repository's `flake.lock`, and invokes sbomnix outside the package's Nix build
sandbox. Main workflow run
[29295978398](https://github.com/choir-hip/go-choir/actions/runs/29295978398),
job `86971008658`, succeeded on that topology in 13m15s on 2026-07-14.

The SBOM job was subsequently paused by `1b28520` through its workflow
condition. Re-enablement therefore means restoring the existing job condition
and proving the already-wired topology again; it does not mean reconnecting or
reimplementing the host-side replacement. SBOM currently runs after `check`
and does not gate `deploy-staging` or FlakeHub publication.
