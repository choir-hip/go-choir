# MissionGradient: Embedded Dolt Runtime Migration v0

Status: local implementation passing; staging landing pending
Date: 2026-05-15
Operator: Codex directly in the repo, not Choir-in-Choir

## Real Artifact

One embedded Dolt workspace per user computer is the canonical durable store for
sandbox product state.

This means **one Dolt workspace/ledger per user computer**, not one Dolt
workspace for VText plus another Dolt workspace for runtime state. VText and
runtime/control tables may be separated by table prefixes or logical SQL
database names inside that same workspace if that proves useful, but they must
not become separate per-user Dolt repositories with separate lineage.

The artifact is not a store interface, a new database driver, a passing unit
test, or a migration script. Those are projections. The real artifact is a
deployed Choir staging system where the active user computer's durable product
truth lives in embedded Dolt rather than a split SQLite-plus-Dolt substrate.

The mission should preserve the current product APIs while moving these current
runtime SQLite facts into the per-user embedded Dolt ledger:

- runs and agent records;
- event and trace records;
- channel messages and inbox deliveries;
- run memory and continuations;
- promotion candidates and run acceptances;
- browser session records;
- researcher findings and worker updates;
- desktop state and desktop workspaces;
- existing VText Dolt state.

The existing VText Dolt workspace is therefore not a sibling to a new runtime
Dolt workspace. The migration should either make the existing VText workspace
the unified per-computer workspace or migrate VText into a new unified workspace
and retire the old VText-only workspace after verification.

Host-side auth and vmctl SQLite are not the target of this mission. They remain
allowed until the platform Dolt layer exists. Filesystem/blob state also remains
outside Dolt when it represents source trees, uploaded/generated bytes, build
artifacts, or materialized aliases.

This is a Codex mission. Do not prompt Choir to perform the implementation.
Choir is part of the deployed product surface to verify after the migration, not
the orchestrator for this substrate change.

## Invariants

- Staging is the acceptance environment for behavior-changing proof.
- The public product APIs keep their current behavior unless a breaking change
  is intentionally named, versioned, and justified.
- Existing user-computer durable facts are not lost during migration.
- VText document history remains version-native in Dolt.
- Runtime events preserve causal order and stream cursors.
- Trace, run acceptance, and promotion APIs continue to be owner-scoped.
- Worker/candidate computers continue to have bounded authority and cannot
  mutate foreground canonical state except through promotion.
- Host auth/session and vmctl routing are not pulled into per-user embedded Dolt
  as part of this slice.
- Large bytes stay in filesystem/blob storage with Dolt metadata, not opaque
  Dolt rows.
- No browser-public internal or test-only route may be used as acceptance proof.
- Node B tracked files are not edited directly as a deployment shortcut.
- Any behavior-changing commit follows the landing loop:

```text
commit -> push origin main -> monitor CI -> monitor staging deploy
-> verify staging commit identity -> run deployed acceptance proof
```

## Value Criterion

Minimize semantic state split inside the user computer while preserving deployed
staging behavior, product API compatibility, restart recovery, traceability,
and rollback.

The loss function penalizes:

- durable product facts still written only to sandbox SQLite;
- duplicate source-of-truth paths after cutover;
- migration paths that cannot be replayed or inspected;
- test-only persistence or local-only proof;
- hidden auth/session, Trace, VText, promotion, or desktop regressions;
- unobserved Node B state after deploy;
- Goodharting by proving only table creation instead of product behavior.

## Quality Gradient

Expected quality: `solid`.

A solid outcome:

- has a clear Dolt-backed runtime store path;
- has migration/import coverage from existing SQLite runtime state;
- preserves the existing runtime API surface;
- removes or quarantines stale SQLite write paths;
- includes product-path staging proof and read-only Node B inspection;
- documents any remaining transitional state with owner, reason, and deletion
  condition.

Substandard work:

- wrapping the existing SQLite file behind a new name;
- keeping SQLite as the hidden writer while Dolt receives partial copies;
- making local tests pass without deployed staging proof;
- moving host auth/vmctl state into the wrong user-computer ledger;
- storing raw blobs in Dolt because it is easier than preserving the blob
  boundary;
- leaving two plausible canonical stores for the same fact;
- creating separate per-user Dolt workspaces for VText and runtime state.

## Homotopy Parameters

Increase realism continuously along these axes:

- read-only inventory of current SQLite/Dolt boundary -> writable Dolt-backed
  runtime store;
- isolated store unit tests -> runtime integration tests -> deployed staging
  acceptance;
- fresh empty unified Dolt workspace -> migration from existing SQLite state;
- single active computer -> active plus worker/candidate computer inspection;
- VText-only Dolt workspace -> one unified sandbox Dolt workspace containing
  both VText and runtime/control state;
- local process restart proof -> Node B VM restart/recovery proof where feasible;
- export-level promotion evidence -> later promotion/publication workflows built
  on Dolt-native records.

At low resolution, a Dolt-backed store may support only the existing runtime
schema with equivalent CRUD behavior. At higher resolution, the store should use
Dolt commits, branches, diffs, and history for meaningful product-state changes.

## Belief State

Current belief:

- docs already define Dolt as canonical product state;
- `internal/store/vtext.go` stores VText documents in embedded Dolt;
- `internal/store/store.go` still stores runtime/control state in SQLite;
- VM sandboxes on staging use `/mnt/persistent/state`, so the immediate Node B
  risk is not `/tmp` durability loss;
- local defaults still use `/tmp` and should be made explicitly ephemeral or
  replaced with a safer local state root;
- promotion/publication work will be cleaner after sandbox product state is in a
  single typed ledger.

Evidence:

- [docs/adr-dolt-as-canonical-state.md](adr-dolt-as-canonical-state.md)
- [docs/architecture.md](architecture.md)
- `internal/store/store.go`
- `internal/store/vtext.go`
- `nix/sandbox-vm.nix`
- Node B review summarized in
  [promotion-consent-layer-research-2026-05-15.md](promotion-consent-layer-research-2026-05-15.md)

Main uncertainties:

- how much SQLite-specific SQL exists in the runtime store beyond DDL, PRAGMA,
  and upsert syntax;
- whether Dolt embedded driver performance/locking is acceptable for event-heavy
  runtime writes;
- whether one SQL database with table prefixes or multiple logical SQL databases
  inside the same Dolt workspace is the cleanest shape for runtime plus VText;
- how much migration compatibility is needed for existing staging computers
  versus a controlled reset of ephemeral QA computers.

Highest-impact uncertainty:

Can the existing runtime store test suite pass against an embedded Dolt backend
without weakening runtime observability or write safety?

Next observation:

Inventory every store method and classify SQL as portable, SQLite-specific, or
Dolt-specific; then run a thin Dolt-backed prototype against the highest-traffic
tables: `events`, `runs`, `channel_messages`, and `vtext_documents`.

## Receding-Horizon Control

Work in short Codex control intervals with bounded mutation radius.

At each interval:

1. state the current boundary being changed;
2. predict the evidence that should change;
3. make the smallest coherent code change;
4. run focused tests;
5. update the belief state if observations surprise the mission;
6. either continue, narrow, branch, rollback, or stop with a precise blocker.

Initial mutation radius:

- do not change frontend UX;
- do not change platform promotion semantics;
- do not change auth/vmctl persistence;
- do not change provider/gateway behavior;
- focus on `internal/store`, runtime store construction, migration utilities,
  and tests.

Widen scope only when store parity is proven.

## Dense Feedback Channels

- store method tests running against SQLite and Dolt during transition;
- migration tests from a populated SQLite runtime DB into Dolt;
- runtime API tests for runs, Trace, VText, promotions, acceptances, desktop

- Dolt workspace inspection showing expected tables and rows;

## Forbidden Shortcuts

- Do not satisfy the mission by updating docs only.
- Do not rename SQLite files or hide them behind a Dolt-flavored interface.
- Do not keep writing product truth to SQLite while copying snapshots into Dolt
  and calling Dolt canonical.
- Do not use browser-public internal routes or test-only endpoints as deployed
  acceptance proof.
- Do not claim local proof for vmctl, worker/candidate computers, auth renewal,
  staging deploy, or Node B persistence.
- Do not edit tracked files directly on Node B.
- Do not weaken docs-only CI path filters to force CI.
- Do not move host auth/vmctl SQLite into per-user Dolt in this mission.
- Do not store large blobs directly in Dolt when the correct artifact is a blob
  hash plus metadata.
- Do not delete old state before migration verification and rollback are clear.

## Rollback Policy

Git rollback:

- every behavior-changing patch must be committed cleanly;
- keep the previous deployed SHA as the immediate platform rollback target;
- if staging regresses, revert the commit or redeploy the previous known-good
  build through the normal deployment path.

State rollback:

- keep a SQLite export/import path during the cutover;
- keep old SQLite runtime state readable until Dolt migration is verified;
- write migration as idempotent or clearly one-shot with backup refs;
- preserve failed Dolt workspaces for diagnosis rather than deleting evidence.

Deployment rollback:

- do not alter CI/CD topology from inside the mission unless the migration
  itself exposes an invariant-level deployment blocker;
- rollback is a normal platform redeploy/revert, not mutation of Node B tracked
  files.

Diagnostic stop:

- before stopping unsuccessfully, use cognitive transforms to look at the
  blocker from storage, security, information-theoretic, and product-operation
  perspectives;
- stop only when the blocker is precise enough to preserve evidence and define
  the next safe probe.

Successful stop:

- after first correctness, do one quality pass for simplification, duplicate
  path removal, tests, docs, and residual-risk clarity.

## Stopping Condition

The mission is complete only when:

- runtime/control product state is written to embedded Dolt in the user computer;
- existing VText Dolt behavior still works;
- migration/import from existing SQLite runtime state is verified or explicitly
  rejected with a safe staging reset policy;
- local tests cover the migrated store boundary;
- behavior-changing changes are committed and pushed to `origin/main`;
- CI and staging deploy are green for the pushed SHA;
- `draft.choir-ip.com/health` reports the deployed SHA;
- deployed product/API or Playwright acceptance passes for VText, Trace,
  promotion candidates, run acceptances, desktop state, and worker delegation;
- read-only Node B review inspects active and worker computer states and confirms
  the product-state ledger shape;
- remaining SQLite stores are only host auth/vmctl or explicitly transitional
  with deletion conditions;
- rollback refs and residual risks are named.

If full cutover is impossible in one run, a partial result is acceptable only if
it reduces semantic split without adding new durable SQLite truth, has a clear
rollback, and leaves the next realism axis sharply defined.
