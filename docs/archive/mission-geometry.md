# Choir Mission Geometry

**Last updated:** 2026-05-29

Choir is a learning system.

It is not a chat app. It is not even a chat-shaped product with a stronger backend. It is not primarily a desktop, a newspaper, a radio station, or a fund either. Those are projections of the same underlying vector. Chat can be an input/control affordance, but the product output is not a discussion thread. Choir is an artifact-native learning system that turns human and machine activity into durable structure: computers, vtexts, claims, sources, tools, agents, compactions, citations, candidate worlds, promotions, and public memory.

The current vector is:

```text
automatic computer -> automatic newspaper -> automatic radio -> automatic capital
```

This is not a roadmap in the ordinary product-management sense. It is a mission geometry. Each term is a projection of the same substrate at a different layer of social and technical reality.

## Center: Choir as learner

Choir is a **statistical, symbolic, and evolutionary learner**.

It is statistical because models infer patterns, retrieve relevance, compress distributions, generalize from examples, and adapt in context.

It is symbolic because Choir stores durable relations: vtexts, claims, sources, citations, versions, obligations, tools, messages, permissions, code, database records, and promotion manifests.

It is evolutionary because background/candidate computers generate candidate worlds, agents mutate them, verifiers and appagents select, promotion retains, and compaction preserves what was learned.

The system learns only when work leaves a durable trace that improves future action. A successful run may promote code, update a vtext, create a tool, add a test, record a source, revise a claim, discover a failed route, or refine a compaction. A failed run can still teach the system if its failure becomes retained structure.

```text
No work without learning trace.
No learning without retention.
No retention without retrieval.
No retrieval without future use.
```

## General science object

The general problem is not "build a better chat app," because Choir is not a chat app. The general problem is:

```text
durable learning control over versioned artifacts
```

A general artifact-native learning control system has:

- persistent computers;
- campaigns;
- mission compiler;
- mission geometry;
- canonical state;
- candidate worlds;
- artifact owners;
- workers;
- messages;
- work orders;
- evidence packets;
- obligations;
- leases;
- verifiers;
- promotion;
- compaction;
- reentry;
- external event feeds;
- sparse high-value human judgment.

Its core objective is to maximize verified artifact improvement over time while minimizing corruption, deadlock, human monitoring burden, and loss of understanding.

That general theory applies to codebases, documents, research projects, media packages, financial models, legal briefs, scientific pipelines, personal knowledge systems, and organizational workflows.

Choir is one concrete product ontology over that science object.

## Automatic computer

The automatic computer is the private agentic workspace. Concretely, a user has a persistent **computer**, not merely a disposable sandbox: a stateful machine-world with VM/runtime state, Dolt/app state, source/build state, blobs, artifact provenance, and route identity.

It is not a chatbot and not an assistant pretending to be a person. It is a durable runtime over artifacts. Agents operate inside bounded roles. The foreground world remains stable. Candidate computers mutate. Canonical state changes only by promotion.

The automatic computer exists to make long-running work possible without forcing the human to monitor every step. The human sets the gradient, supplies judgment, corrects the system, and approves irreversible transitions. The system handles context routing, compaction, candidate-world mutation, verification, and promotion.

This is the substrate layer. If this layer is wrong, everything above it becomes slop.

Read [computer-ontology.md](../computer-ontology.md) for the ledger split between VM/runtime, Dolt/app state, source/build state, blobs, artifact provenance, and routes. The key distinction is that users can evolve their own computers quickly without global platform deploys, while shared/platform changes require typed artifacts and higher-ceremony promotion.

## Campaign compiler

Campaign Compiler is the control layer that turns intent and signals into
bounded, evidence-bearing work across computers, agents, runtimes, users, and
perspectives.

It is not a better prompt format. It is not a Codex `/goal` upgrade. It is the
Choir-native layer that keeps durable campaigns coherent while many missions
start, stop, fail, promote, roll back, and reenter the human.

The durable object is:

```text
Campaign
-> compiled MissionGeometry
-> WorkOrders
-> candidate/computer/runtimes
-> EvidencePackets
-> VText / Trace updates
-> promotion or rollback
-> ReentryDigest
-> campaign memory
```

Campaign Compiler should select cognitive transforms as part of compilation.
Transforms are not decorative lenses. A transform is retained when it changes
the route, verifier, evidence plan, scope, capability envelope, or stopping
condition.

This is the bridge from automatic computer to automatic newspaper. A newspaper
is not a single goal; it is a standing campaign over public evidence streams,
source histories, claims, corrections, publications, and reentries.

## Automatic newspaper

The automatic newspaper is the public memory layer.

It is not AI summaries of news. It is a living artifact graph: vtexts, sources, claims, corrections, citations, track records, prior art, public speech, and future relevance.

The newspaper makes thought citeable, disputable, forkable, correctable, and reusable. It lowers the cost of discovering what happened, who said what before, which sources aged well, which claims survived, and which prior work matters now.

The automatic newspaper turns private artifacts into public memory.

## Automatic radio

The automatic radio is the embodied traversal layer.

It renders the artifact graph into time. The user can listen while walking, driving, cooking, cleaning, resting, or thinking. They can interrupt, ask, clarify, disagree, source, save, branch, or publish.

Radio is not a pivot away from vtext. Radio depends on vtext. **Vtext is the score; radio is the performance.**

AI narration organizes. Human voice testifies. No voice cloning. Real speech remains evidence. The radio is not a synthetic companion. It is a source-grounded traversal of public and private artifacts.

The automatic radio turns public memory into lived attention.

## Automatic capital

Automatic capital is the capital-formation layer.

It is not the immediate priority, and it should not be mistaken for "start a fund." It is the recognition that the same learning system that identifies future-relevant ideas can also identify underpriced agency.

If Choir can see who was early, who corrected well, who built useful tools, whose claims aged well, whose artifacts became dependencies, and whose work becomes useful to future discourse, then Choir is already producing underwriting signal.

Capital is downstream of signal. Automatic capital is what happens when a learning system routes money, compute, tools, distribution, and credibility toward future-relevant agency before existing markets have priced it.

The phrase is:

```text
Choir IP -> Choir LP
```

Choir IP means future-useful thought becomes an asset through provenance, citation, and dependency.

Choir LP means future-useful agency can receive capital through the same signal field.

This does not mean Choir should rush to build a fund. It means automatic capital is on the same vector as the automatic computer, automatic newspaper, and automatic radio. It is a later projection of the same substrate.

## Vtext as semantic control surface

`vtext` should be designed less like a document and more like a canonical semantic control surface.

It has visible text, but also claims, sources, clips, obligations, transforms, unresolved tensions, update hooks, render targets, publication state, and provenance. The screen version is one view. Radio is another view. Public newspaper is another. Agent work is another.

The next specification target is:

```text
Define vtext as the canonical versioned semantic artifact that coordinates evidence,
obligations, transformations, and projections, including screenless radio.
```

## Mission gradient

The mission is to build a learning system that converts human and machine activity into durable, correctable, citeable, and useful artifacts.

The local gradient is:

- preserve canonical state;
- mutate candidate worlds;
- promote only verified deltas;
- retain learning traces;
- make correction valuable;
- route attention toward sources and claims;
- turn artifacts into public memory;
- turn public memory into radio traversal;
- let future relevance reveal where capital should flow.

The center is learning. The vector is automatic computer -> automatic newspaper -> automatic radio -> automatic capital.

The work now is to build the substrate well enough that later projections become obvious rather than forced.
