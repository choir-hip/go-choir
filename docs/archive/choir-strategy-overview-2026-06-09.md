# Choir Strategy Overview

Date: 2026-06-09

Status: current strategic orientation

This document consolidates the current high-level strategy after the Wire
ontology correction. It draws on founder strategy sessions and product learning
but is written as repo-native context, not copied pitch copy.

## One-Line Frame

Choir helps organizations own their AI cloud.

## Deeper Frame

Choir helps organizations own the learning generated by their work.

Models churn. Organizational learning compounds. The valuable layer is not only
the answer returned by a model; it is the durable context, sources, corrections,
versions, traces, style, evals, model-routing choices, and publication history
created while doing work. Choir keeps that compounding layer under the user's or
organization's control.

## Product Object

Choir is not a chatbot and not Notion-with-a-bot. The product object is a
persistent computer: a durable machine-world where documents, agents, files,
source artifacts, model policy, app state, candidate work, and publication
records can evolve over time.

Chat remains useful as input, but chat history is not the work system. Durable
artifacts are.

## Strategic Vector

```text
private AI cloud
  -> Wire
  -> radio
  -> later capital/economic surfaces
```

The older shorthand was automatic computer -> automatic newspaper -> automatic
radio -> automatic capital. The current language keeps the same vector but
separates the reusable substrate:

- **Private AI cloud**: persistent computers and private source-backed work.
- **Wire**: source-to-VText substrate for public and private source synthesis.
- **Radio**: voice/screenless traversal and supervision over VTexts, sources,
  and agent state.
- **Capital/economic surfaces**: later consequences of durable citation,
  provenance, contribution, and compute accounting.

## Why Not Just GPT, Claude, Codex, Or Copilot?

Use them. They are powerful models and agent surfaces.

Choir is the customer-owned layer above models: private data, durable artifacts,
source memory, model routing, run state, role-specific agents, publications,
candidate computers, verifier evidence, and organizational learning. The
business should rent compute and own intelligence.

## First Wedge

The first commercial wedge is private legal/professional cloud deployment:

- client and matter files;
- source-backed memos;
- style-aware drafts;
- durable matter memory;
- client-ready artifacts;
- private deployment on infrastructure the client controls.

The wedge is services-led but not bespoke-only. Each engagement should harvest
requirements and produce reusable primitives for the repo: source connectors,
document extraction, VText workflows, publication policy, private-cloud
deployment patterns, and role-specific edition/briefing behavior.

## Community Cloud And Private Clouds

Choir should support multiple clouds, not just multiple users in one shared app.

```text
Choir Community Cloud
  public shared deployment at choir.news
  public Wire
  public publication graph
  public user computers

Private Choir Cloud
  client-owned NixOS host or host cluster
  private platform computer(s)
  many private user computers
  candidate computers
  private source systems
  controlled publication/subscription boundaries
```

The Community Cloud and Private Clouds use the same core technology stack:
NixOS hosts, persistent computers, VText, source artifacts, processors,
reconcilers, researchers, VText agents, candidate computers, publication, and
indexes as caches.

They differ by ownership, visibility, source connectors, egress policy,
compliance constraints, and publication rights.

## Wire

Wire is the reusable source-to-VText substrate.

The Community Cloud's public Wire ingests public sources and produces public
source artifacts, article/report VTexts, and edition VTexts.

Private Clouds run private Wire instances over private data sources, subscribed
public Wire artifacts, and domain-specific sources. The same substrate can
become a legal Matter Wire, biotech Research Wire, executive briefing Wire,
Market Wire, Tech Wire, Science Wire, or internal risk Wire.

## Personalization

Personalization is userland authorship, not centralized recommendation.

Platform Wire produces shared canonical source-backed VTexts. User computers
run user-level processors and reconcilers over accessible public/private
corpora, then ask user VText agents to create user-owned editions, briefings,
forks, notes, and alerts.

In a private cloud, the CEO, VP of engineering, junior designer, and HR manager
can all consume the same firm/public corpus while receiving different
user-owned VTexts.

## VText As Control Plane

VText is versioned semantic state. Users steer work by editing durable artifacts:
adding a source, revising a line, changing a requirement, accepting a version,
or publishing a revision.

This is the category demonstration: the same prompt in a chat app yields a
buried answer; in Choir it yields a cited, versioned, editable, publishable
artifact.

## Voice And Radio

Radio is not a separate podcast app. It is the voice/screenless endpoint over
the same persistent source/VText/agent system.

The intelligence stays in the owned middle: the user's or organization's
computers, sources, VTexts, and agents. Voice models are endpoints: ears and
mouth. Radio becomes useful when it can traverse source-backed VTexts, interrupt
for decisions, and let the user redirect long-running work away from the
screen.

## Open Source And Business Model

The core should remain open source and self-hostable. Revenue comes through
deployment, customization, managed private clouds, support, and reusable modules
discovered through service work.

Services are the wedge, not the identity. The durable asset is the reusable
owned-learning substrate.
